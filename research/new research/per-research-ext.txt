Short answer: on modern phones you should plan for **~30–40 % of physical RAM as a *safe* budget** for your React Native + llama.rn process if you want to avoid random kills. That makes a single 350M Q4_K_M model + RN app realistic on 4 GB devices, but a 1.2B Q6_K at 8K context is only comfortable on 6 GB+ and even then is tight.

Below are realistic ranges and trade‑offs per question.

***

## 1. iOS per‑app memory budget (Jetsam)

Apple doesn’t publish hard per‑app limits; the OS uses **Jetsam** to kill processes when total system memory gets tight. Empirical measurements on earlier devices show apps crash when they reach roughly **50–70 % of physical RAM**.[1][2]

Older measurements (but still useful for ratios):  

- iPhone 5 / 1 GB RAM → app could allocate ~645 MB (≈62 %).[2]
- iPhone 5s / 1 GB RAM → ~646 MB (≈63 %).[1]
- iPad Pro 12.9 / ~4 GB RAM → ~3.0 GB available to one app (≈77 %).[2]

Extrapolating to your targets:

- **iPhone 13 (4 GB)**  
  - Realistically: plan for **1.5–2.0 GB steady‑state** for your app.  
  - You *might* get closer to 2.5–3 GB in ideal conditions, but background processes and OS daemons will often push Jetsam to kill you sooner.  

- **iPhone 15 (6 GB)**  
  - Realistically: **2.5–3.0 GB** is a safe budget, **3.5–4 GB** is probably near the cliff.  
  - Above that, you’re competing heavily with SpringBoard, Safari, Photos, etc., and background apps will be aggressively killed to keep you alive.

So for planning: assume **~40 % of total RAM as a conservative ceiling** (≈1.6 GB on 4 GB, ≈2.4 GB on 6 GB) if you want to survive in worst‑case multitasking, and maybe **60–70 %** as an upper bound in controlled conditions.[1][2]

***

## 2. Android per‑app memory budget

Android uses a **heap limit** per process, exposed via `ActivityManager.getMemoryClass()` and `getLargeMemoryClass()`.[3][4]

- `memoryClass` returns the “approximate per‑application memory class of the current device” in MB; you should treat this as your Java/Kotlin heap budget.[4]
- `largeHeap` in the manifest can raise this for memory‑intensive apps, but you shouldn’t rely on it unless absolutely necessary.[3]

Rule of thumb today:

- **4 GB Android device**  
  - `memoryClass` often in the **256–384 MB** range;  
  - `largeHeap` can raise you into the **512–768 MB** range, depending on OEM.[4][3]
  - Native memory (C++ / llama.cpp) is *in addition* to this but still counts against overall system RAM; the LM runtime tends to allocate big native heaps, so you can OOM even if the Java heap is fine.

- **6 GB Android device**  
  - `memoryClass` more commonly **384–512 MB**, `largeHeap` up to **1 GB+** on some devices.  
  - Practically, you can often use **1.5–2.0 GB total (Java + native)** before LMK starts killing you, but this varies a lot by OEM and background load.

TL;DR: you **must** measure `memoryClass` at runtime and log real memory use on your target devices. Use `Runtime.getRuntime().maxMemory()` and `Debug.getNativeHeapAllocatedSize()` to watch both Java and native usage.[3][4]

***

## 3. Baseline React Native + Hermes memory

There are some published comparisons between JSC, Hermes, and V8:

- A profiling study found Hermes had **initial heap ~30 MB** and **peak usage ~136 MB** for a moderately complex RN app, versus JSC starting around 50 MB and peaking closer to 200 MB under the same scenario.[5][6]

In practice, for a non‑trivial app:

- **JS heap (Hermes)**:  
  - Idle: **30–60 MB**.  
  - Normal use (navigation, moderate state, some lists): **80–150 MB**.[5]
- **Native RN + UI + OS allocations**:  
  - Typically another **50–150 MB** depending on screens, images, and libraries.  

So a realistic baseline for **RN + Hermes** with no model loaded is:

- **Low‑end**: ~100 MB.  
- **Typical real app**: **150–250 MB** total (JS + native), before any llama.rn context.

Plan for at least **200 MB** baseline if you’re shipping something non‑trivial.

***

## 4. llama.rn memory usage estimates

`llama.rn` is a thin binding over llama.cpp. The README recommends matching llama.cpp flags: you set `n_ctx`, `n_gpu_layers`, etc., and it allocates model weights + KV cache accordingly.[7]

Community guidance for llama.cpp is:[8][9][10]

- **Weights**: roughly equal to the **GGUF file size**, plus overhead; on CPU RAM this is often **1.0–1.3× file size**.[9]
- **KV cache**: depends on layers, head dimension, context length and dtype; a measured rule‑of‑thumb is **≈0.75 MB per layer per 1K tokens (FP16)**, and far less if you quantize the KV cache.[10]
- For 8B models at 4K context, KV alone is ~200 MB; for 8K, ~400 MB.[8]

Given that, we can estimate:

### 4.a 350M Q4_K_M, context 1024

Typical 350–360M Q4_K_M GGUF files are about **0.2–0.3 GB**.[11][12]

- **Weights in RAM**:  
  - ~0.25 GB file → **0.25–0.35 GB actual** in RAM (including extra buffers).[9]
- **KV cache @ 1K**:  
  - 350M models have far fewer layers and heads than 7B. Scaling from 8B numbers (200 MB KV at 4K) gives you **~10–20 MB KV at 1K** for a 350M model (rough order of magnitude).[10][8]
- **Total llama.cpp / llama.rn footprint** (weights + KV + scratch):  
  - **≈300–450 MB** is a realistic band.  

Add that to your RN baseline (~200 MB) and you’re looking at **~500–650 MB total app memory** with a 350M Q4_K_M at 1K context.

This is entirely feasible on:

- **iPhone 13 (4 GB)** if the device isn’t heavily loaded.  
- **4 GB Android** if you’re careful with largeHeap and don’t have many other native buffers.

### 4.b 1.2B Q6_K, context 8192

We don’t have a specific 1.2B Q6_K GGUF example in the search, but:

- 1B‑class Q6_K GGUF files tend to be around **2–3 GB**.[13][14]
- For rough planning, assume **2.5 GB file size**.

Then:

- **Weights in RAM**:  
  - ~2.5 GB file → **2.5–3.2 GB** RAM usage.[9]
- **KV cache @ 8K**:  
  - Using the 8B example: 8B at 8K → ~400 MB KV.[8]
  - 1.2B is roughly 1.2 / 8 ≈ 0.15 of that, so **~60 MB KV cache at 8K** in FP16; quantized KV reduces further.[15][10]
- **Total llama.cpp / llama.rn footprint**:  
  - **≈2.8–3.5 GB** is a realistic range.

Add RN baseline (~200 MB) and you’re at **~3.0–3.7 GB** for the whole app.

This is:

- **Marginal or impossible on 4 GB devices** – Jetsam/LMK will often kill you once you get close to 3 GB usage.[16][2]
- Only realistic on **6 GB devices or better**, and even there you’re sharing with the rest of the system.

### 4.c Both models loaded simultaneously

If you keep **both contexts fully loaded**:

- 350M Q4_K_M @ 1K context: ~0.3–0.45 GB.  
- 1.2B Q6_K @ 8K context: ~2.8–3.5 GB.  

Total **models alone**: **~3.1–4.0 GB**, plus **~0.2 GB RN baseline** → **~3.3–4.2 GB**.

That is:

- Unrealistic on any **4 GB** device.  
- Very risky even on **6 GB** iPhones—it would leave almost no headroom for the OS; Jetsam will likely kill something.[16][2]
- Only viable on **8 GB+ iPad / high‑end Android tablets**, and even then you’d want to reduce context or unload models when not in use.

**Design implication:** on phones, you should generally **load a single model context at a time**, and consider either:

- One “main” 350M model that does everything; or  
- Dynamic model swapping (unload one context, load another) for rare flows.

***

## 5. Multiple llama.rn contexts at once

`llama.rn` itself does not forbid multiple contexts; its API is just:

```ts
const context = await initLlama({ model, n_ctx, n_gpu_layers, ... })
```

You can call `initLlama` multiple times and keep multiple `context` objects in JS memory.[7]

However:

- **Memory scales linearly** with each context’s weights + KV cache. There’s no internal sharing of weights or KV between contexts.  
- For concurrency, `llama.rn` already provides a **slot‑based parallel decoding API** (`context.parallel.enable({ n_parallel, n_batch })`) that shares a single KV cache across slots.[7]
  - This is almost always more memory‑efficient than creating multiple separate contexts for multiple conversations.[7]

So:

- **Yes, you *can* load two contexts at once**, but on mobile you should treat that as an advanced/edge scenario and measure RAM carefully.  
- The more idiomatic pattern is: **one context per model**, use `n_parallel` slots for concurrent requests; avoid multiple models unless you’re on 8 GB+ devices.[7]

***

## 6. Metal GPU sharing for multiple models on iOS

`llama.rn` uses the same Metal backend as llama.cpp and notes:[7]

- Metal requires at least an **Apple7‑class GPU** and is not available in the simulator.  
- Extended Virtual Addressing and **Increased Memory Limit** capabilities are recommended for iOS projects, implying heavy unified‑memory usage.[7]

Key points:

- Apple GPUs use **unified memory** – GPU and CPU share the same physical RAM pool.[17]
- Each llama.cpp context with `n_gpu_layers > 0` allocates Metal buffers for the offloaded layers. Multiple contexts or models will **each allocate their own GPU buffers**, all coming from the same unified RAM.  
- Metal itself allows multiple `MTLDevice` and `MTLCommandQueue` users in one process, so there’s no hard “one model only” restriction – but you are bound by RAM: the sum of all Metal allocations + CPU allocations must fit under Jetsam limits.[17][16]

Practically:

- **Yes**, you can run multiple llama.rn contexts using Metal on iOS, but:
  - Their GPU memory usage **adds up**; there’s no automatic weight sharing across models.  
  - Unified memory still counts against the same per‑app Jetsam budget as CPU allocations, so moving weights to Metal **does not magically give you extra RAM**. It just reallocates within the same pool, sometimes with better bandwidth.[16][17]

For a phone‑class device, a good pattern is:

- Offload most or all layers of **one model** (e.g., 350M) to Metal.  
- Keep context size moderate (1K–2K) and use parallel slots for concurrency.  
- Avoid multiple Metal‑backed contexts unless you’ve verified you stay well below ~2–3 GB total on a 6 GB device.

***

## Practical budgets to design around

Putting it all together for your React Native + llama.rn design:

- **4 GB iOS/Android phones**  
  - Safe app budget: **<=1.5 GB**.  
  - RN + Hermes: ~0.2 GB.  
  - Leaves **~1.3 GB** for models.  
  - Realistic: **one 350M Q4_K_M** with 1K–2K context, possibly a small embedding model *instead of* a second LLM.

- **6 GB phones (iPhone 15, many Android flagships)**  
  - Safe app budget: **~2–2.5 GB**.  
  - RN + Hermes: ~0.2 GB.  
  - Leaves **~1.8–2.3 GB** for models.  
  - Realistic: one **350M–1B Q4/Q5** model at modest context, or **1.2B Q6_K** at 8K context if you’re okay living near the edge and aggressively shutting down other heavy processes.
