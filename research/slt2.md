# Path A vs. Path B: A Decision Framework for On-Device Intent Classification in React Native

This report provides a comprehensive deep-dive analysis into the evaluation of MobileBERT and alternative lightweight classification models against an existing 350-million-parameter language model for on-device intent classification within a React Native and Expo application. The research goal is to identify the optimal path forward for Lyla, a private AI assistant, by comparing two distinct strategies: leveraging an already-loaded general-purpose model (Path A) versus integrating a purpose-built, fine-tuned classifier (Path B). The analysis rigorously assesses each path against critical constraints, including model size (<50MB), inference latency (<50ms), accuracy (>95%), and integration complexity within the specified technology stack. The findings are based exclusively on the provided research materials, ensuring fidelity and traceability for every factual claim.

## Comparative Analysis of Existing Model Prompting (Path A)

The first strategic path, designated Path A, involves utilizing the LFM2.5-350M model already loaded within the application for intent classification tasks. This model, described as a family of hybrid models designed for efficient on-device deployment and strong task capabilities, is presented as a powerful asset that incurs no additional memory cost for its use [[11](https://modelscope.cn/models/onnx-community/LFM2.5-350M-ONNX), [36](https://arxiv.org/abs/2511.23404)]. The core mechanism for this path is the use of natural language prompts to direct the model to perform a classification task, effectively turning a general-purpose conversational AI into a specialized intent classifier. For example, a user would send a query structured as "Classify the following text into one of these intents: [list of 20+ intents]. Text: '...'" to the model. This method falls under the category of zero-shot or few-shot text classification, where the model performs a task without having been explicitly trained on labeled examples for that specific task [[14](https://openreview.net/pdf/edce3377abde61c6b55554588ca7cee68f38ce18.pdf)]. While this approach offers immediate feasibility, its success is contingent upon overcoming significant uncertainties related to accuracy and latency, which are the most critical factors for user experience and product viability. The primary advantage of Path A is its zero marginal cost in terms of memory and development time; since the model is already part of the application's assets, there is no need for new library integrations, native module development, or complex build configurations [[11](https://modelscope.cn/models/onnx-community/LFM2.5-350M-ONNX), [41](https://www.modelscope.cn/models/mlabonne/LFM2.5-350M-GGUF)]. This makes it the lowest-risk option from an engineering perspective. However, the trade-offs in performance and reliability must be carefully scrutinized against the project's stringent requirements of >95% accuracy and sub-50ms latency. The fundamental question for Path A is whether a general-purpose model can be reliably coaxed into achieving the precision of a purpose-built classifier through clever prompting alone.

A key strength of Path A lies in its flexibility and adaptability. Because the classification logic is encoded in the prompt rather than being hard-coded into the model's weights, adding new intents should theoretically be a simple matter of updating the list of possible labels within the prompt string. This dynamic capability could be highly advantageous for a rapidly evolving application like Lyla, allowing for quick adjustments to the intent set without requiring a full model retraining and redeployment cycle. This contrasts sharply with Path B, where adding new classes would necessitate either fine-tuning the existing model or training a new one, a process that is more time-consuming and resource-intensive. Furthermore, the LFM2.5-350M model itself is positioned as a best-in-class performer, rivaling much larger models and delivering fast edge inference speeds, with reported figures of 313 tokens per second on an AMD CPU [[41](https://www.modelscope.cn/models/mlabonne/LFM2.5-350M-GGUF)]. This suggests that the underlying computational engine is robust and optimized for on-device execution. The model architecture, derived from the LFM2 family, is explicitly designed for efficiency and strong task capabilities, making it a compelling foundation for a wide range of applications beyond simple conversation [[35](https://arxiv.org/html/2511.23404v1), [36](https://arxiv.org/abs/2511.23404)]. However, the very fact that it is a "hybrid" model optimized for broad utility raises a critical point of uncertainty regarding its suitability for the highly structured problem of multi-intent classification [[11](https://modelscope.cn/models/onnx-community/LFM2.5-350M-ONNX)]. General-purpose models excel at understanding nuanced context and generating human-like text, but their performance on discrete, categorical classification can be inconsistent and unpredictable compared to models specifically architected and trained for that purpose.

The most significant drawback of Path A is its reliance on prompt engineering, a discipline fraught with challenges and lacking the deterministic guarantees of supervised learning. The accuracy of the classification will be heavily dependent on the precise wording of the prompt, the order of the intents listed, and the inherent biases of the base model. Research indicates that optimizing prompts for boundary cases—utterances that are ambiguous or lie between multiple intents—is a non-trivial process that requires iterative generation of synthetic data and careful calibration [[26](https://openreview.net/pdf/6dcd20894c89640c9bcb98ace592edf0276b9363.pdf)]. This is not a one-time setup but an ongoing optimization effort. Achieving the target of >95% accuracy is not guaranteed; general-purpose language models often struggle with the zero-shot setting, and performance can vary significantly across different datasets and tasks [[27](https://www.researchgate.net/publication/390109492_From_Large_to_Mammoth_A_Comparative_Evaluation_of_Large_Language_Models_in_Zero-Shot_Vulnerability_Detection)]. The BTZSC benchmark, which evaluates zero-shot text classification across 22 public datasets, highlights the variability in model performance, underscoring the difficulty of relying solely on prompting for high-stakes classification tasks [[15](https://arxiv.org/html/2603.11991), [37](https://arxiv.org/abs/2603.11991), [38](https://arxiv.org/pdf/2603.11991)]. Therefore, while Path A promises a rapid and inexpensive implementation, it introduces a substantial element of risk concerning the primary metric of success: accuracy. The team would be betting that the LFM2.5-350M model, when guided by a well-crafted prompt, can consistently outperform the current 60–70% accurate regex system and meet the >95% threshold.

Latency presents another area of concern for Path A. The estimated latency of 50–100ms per query is a crucial data point that directly impacts the user experience. While this range may be acceptable for many background operations, it pushes the upper limit of what is considered instantaneous interaction. For a real-time assistant like Lyla, where users expect near-immediate responses, latencies approaching 100ms can introduce noticeable lag. The target of <50ms is a common benchmark for creating a fluid and responsive UI, and exceeding it could lead to a perception of the application being slow or unresponsive [[1](https://arxiv.org/html/2601.03290v1)]. The actual latency will depend on several factors, including the length of the input text, the complexity of the prompt, the load on the device's CPU/GPU, and the efficiency of the runtime environment (Hermes). The fact that the LFM2.5-350M model achieves high token decoding speeds suggests that the raw computational power is there, but the overhead of parsing the prompt and generating a structured output adds to the total processing time. This latency window also includes the ~50–100ms mentioned for the prompt-based approach, which must be accommodated within the overall application performance budget. In contrast, a dedicated classifier like MobileBERT is projected to have inference times well under this threshold, potentially in the 10-20ms range, offering a smoother and more immediate user experience [[1](https://arxiv.org/html/2601.03290v1)]. This difference in responsiveness is a key competitive advantage that Path B offers, albeit at a much higher implementation cost.

In summary, Path A represents a pragmatic, low-friction strategy that leverages an existing, powerful asset. Its strengths are clear: zero memory overhead, immediate implementation potential, and operational flexibility for managing intents. However, these benefits come with significant risks. The central unknown is whether the model's accuracy can be reliably driven above 95% through prompting, a challenge that is not guaranteed and requires extensive empirical validation. The latency, while likely acceptable, may not provide the optimal user experience and approaches the defined upper limit. This path is therefore best viewed as a proof-of-concept test. Before committing to the higher complexity of Path B, it is imperative to rigorously measure the performance of Path A on the specific 20+ intent dataset that will be used in production. Only if Path A fails to meet the critical thresholds for accuracy and/or latency should the team proceed to explore the more complex but potentially more performant solution offered by Path B. This risk-averse approach allows for an evidence-based decision, ensuring that the significant engineering investment in Path B is only made when absolutely necessary.

## Evaluation of Lightweight Models for On-Device Deployment (Path B)

Path B proposes a departure from the general-purpose model-by-prompting approach and instead advocates for the integration of a purpose-built, lightweight classification model. This path aims to achieve superior performance in both accuracy and latency by selecting a model architecture specifically optimized for on-device inference and then fine-tuning it on the target dataset of 20+ custom intents. This strategy moves away from the inherent unpredictability of zero-shot prompting and toward the proven reliability of supervised learning. The core components of this path involve selecting an appropriate model candidate, exporting it to a suitable on-device format, and integrating it into the React Native application via a compatible inference engine. This route promises to deliver a solution that is tightly optimized for the classification task, but it comes with a significant increase in technical complexity and data management requirements. The choice of model is paramount, as it dictates the fundamental trade-off between size, speed, and accuracy. The provided research points to several strong contenders within the BERT family and a compelling alternative in FastText.

Among the Transformer-based models, MobileBERT emerges as a particularly strong candidate. It was explicitly designed for mobile environments, achieving a GLUE score of 77.7 with only 25.3 million parameters—a fraction of the 110 million in a standard BERT-base model—while maintaining competitive accuracy [[1](https://arxiv.org/html/2601.03290v1)]. Its design philosophy prioritizes efficiency, making it an ideal fit for the stringent memory constraints of the project. Another key piece of evidence supporting its viability comes from its successful application in a client-side phishing detection system, where it achieved the highest accuracy (0.96) among tested small language models, demonstrating its effectiveness in a real-world on-device classification task [[10](https://arxiv.org/html/2509.10858v1)]. While its benchmarked inference latency on a Pixel 4 phone was 62ms, this figure was likely generated using a full-precision FP32 model. Modern mobile GPUs, coupled with optimized inference engines like those found in the Hermes JavaScript engine, could significantly reduce this latency [[5](https://arxiv.org/pdf/2508.18255)]. The research also notes that for interactive UI applications requiring a 100ms response time, architectures like MobileBERT are viable options [[1](https://arxiv.org/html/2601.03290v1)]. Other notable variants include DistilBERT and TinyBERT. DistilBERT, with 66 million parameters, offers 60% faster inference than BERT-base but has a reported latency of 230ms, which seems high and may reflect older hardware or software [[1](https://arxiv.org/html/2601.03290v1)]. Nonetheless, it has demonstrated strong performance in intent classification benchmarks, outperforming other models like Phi2 and LLAMA3 [[30](https://www.academia.edu/125096234/INTENT_RECOGNITION_USING_DISTILBERT_AND_LANGUAGE_MODELS)]. TinyBERT offers a family of sizes, with a 4-layer version (14.5M parameters) achieving over 96.8% of BERT-base's performance, though it shares a similar 62ms latency with MobileBERT [[1](https://arxiv.org/html/2601.03290v1), [23](https://arxiv.org/abs/1909.10351)]. The existence of these various models provides a spectrum of choices, allowing the development team to select the optimal balance between parameter count, latency, and accuracy for their specific needs.

However, the assumption that a Transformer-based model is the only or even the best solution is challenged by the evidence for FastText. This non-transformer, bag-of-words style model with subword information has shown remarkable performance on classification tasks. One study detailed a proposed CNN + FastText architecture that achieved a classification accuracy of 97.26% on a legal document dataset, surpassing even a fine-tuned BERT model (97.12%) [[2](https://arxiv.org/html/2604.17674v1)]. The implications of this finding are profound for a resource-constrained environment. The FastText model achieved this high accuracy with only 5.1 million parameters, a fraction of the ~25M required by MobileBERT. More impressively, its inference latency was measured at just 0.31 ms per document, over 13 times faster than the BERT model's 4.25 ms [[2](https://arxiv.org/html/2604.17674v1)]. This combination of extremely high accuracy potential and exceptional computational efficiency makes FastText a formidable contender. Its ability to handle domain-specific vocabulary and out-of-vocabulary words through its subword decomposition mechanism is a significant advantage, as it can create meaningful representations for rare or unseen terms without needing extensive hand-crafted lexicons [[2](https://arxiv.org/html/2604.17674v1)]. Furthermore, the model demonstrated robustness to input perturbation, maintaining high accuracy even when noise was added to the embeddings, indicating a level of stability that is desirable in a production environment [[2](https://arxiv.org/html/2604.17674v1)]. Given these attributes, FastText warrants serious consideration as a baseline model. It may provide the desired performance with a fraction of the implementation complexity and resource consumption of a full-fledged Transformer model. A direct comparison on the user's specific dataset would be the definitive step, but the preliminary evidence strongly suggests that a simpler model might be preferable if it can meet the accuracy requirements.

Ultimately, the selection of a model for Path B is a strategic decision. While Transformers like MobileBERT offer state-of-the-art contextual understanding, their overhead may be unnecessary for a well-defined classification task. FastText's demonstrated ability to match or exceed Transformer performance with a fraction of the resources presents a powerful alternative. The recommended approach would be to begin with FastText as a first-principles baseline due to its simplicity and efficiency. If its accuracy proves insufficient for the project's >95% requirement, then a more complex Transformer model like MobileBERT should be adopted. This tiered approach minimizes initial development effort while still allowing access to the highest levels of performance if needed. Regardless of the final choice, the core principle of Path B remains: to build a dedicated, finely tuned classifier that is engineered to meet the specific demands of the intent classification task, thereby providing a more reliable and performant solution than the general-purpose model used in Path A.

## Integration Feasibility and Engineering Complexity of Path B

While Path B promises superior performance, it introduces a significant increase in engineering complexity that must be carefully managed. Integrating a new, lightweight model like MobileBERT or FastText into a React Native and Expo application is not a trivial task. It requires moving beyond the pure-JavaScript ecosystem and engaging with native code, which increases the maintenance burden and deviates from the "write-once-run-anywhere" ethos of React Native. The primary challenges revolve around selecting a compatible inference engine, managing native module dependencies, and navigating the complexities of Expo's build system. These hurdles represent a substantial barrier to entry and require a dedicated allocation of engineering resources for successful implementation.

The first step in integrating a model is choosing an appropriate inference engine that is compatible with React Native and can run on both iOS and Android. The two dominant formats for on-device deployment are ONNX (Open Neural Network Exchange) and TFLite (TensorFlow Lite). ONNX is an open-standard format that allows models from different frameworks (like PyTorch or TensorFlow) to be interoperable. To run an ONNX model in a React Native app, developers can use libraries such as `onnxruntime-react-native`. This library provides a bridge between JavaScript and the ONNX Runtime C++ core, enabling model execution on the device. TFLite is Google's framework for deploying TensorFlow models on mobile and edge devices. For TFLite models, several React Native libraries exist, including `react-native-fast-tflite`, which is designed for high-performance inference, and the more general-purpose TensorFlow.js, which can also load and run TFLite models [[3](https://www.linkedin.com/pulse/real-time-image-classification-tensorflow-daniel-wind), [4](https://dev.to/amanhimself/image-classification-on-react-native-with-tensorflow-js-and-mobilenet-dpc)]. The choice between ONNX and TFLite often depends on the toolchain used for fine-tuning the model. Many researchers and practitioners use PyTorch, which exports well to ONNX. Others prefer TensorFlow, making TFLite the natural choice. The selected model (e.g., MobileBERT) must be exported to one of these formats before it can be integrated into the application. This process involves writing scripts to convert the trained model weights and architecture into the chosen portable format.

A critical misconception to address is the feasibility of running MobileBERT through `llama.rn`. `llama.rn` is a specific React Native wrapper around the `llama.cpp` library, which is designed to efficiently run LLMs distributed in the GGUF (Georgi Gerganov Unified Format) [[1](https://arxiv.org/html/2601.03290v1)]. GGUF is tailored for large language models trained for generative tasks and general reasoning. MobileBERT, however, is a Transformer encoder model specifically designed for sequence classification tasks. It has a fundamentally different architecture, input/output structure, and parameter layout compared to the decoder-only models that `llama.cpp` is built to handle. Attempting to force a MobileBERT model into a GGUF-compatible format and run it on `llama.rn` would almost certainly fail. The model would not be compatible with the `llama.cpp` runtime, which expects a specific tensor layout and execution flow for autoregressive generation. Therefore, running MobileBERT requires a separate, dedicated runtime like ONNX Runtime or TensorFlow Lite, not the `llama.rn`/`llama.cpp` ecosystem. This distinction is crucial because it clarifies that adopting Path B does not leverage the existing infrastructure used for the 350M model in Path A, but rather requires building a new, parallel inference pipeline.

The integration process itself typically involves creating and linking a native module. Even with wrappers like `onnxruntime-react-native`, some native code compilation is usually required during the build process. When using Expo's managed workflow, adding native dependencies can be problematic. The managed workflow abstracts away the native project files (the `/ios` and `/android` directories), making it difficult to add arbitrary native libraries. To overcome this, developers often need to "eject" to a bare workflow, which exposes the native code and gives full control over the build process. However, ejecting is a significant commitment, as it limits future access to new managed workflow features. A more modern and flexible approach within the Expo ecosystem is to use Expo Application Services (EAS). EAS Build can take a managed Expo project and compile it into a bare React Native project, applying any necessary native modifications (like adding the ONNX Runtime library) before building the final `.ipa` and `.apk` files. This allows teams to maintain the convenience of the managed workflow while still being able to integrate complex native dependencies. This process, however, adds another layer of configuration and potential failure points to the CI/CD pipeline. The engineering effort includes not only writing the JavaScript/TypeScript bindings but also configuring the native build settings, handling dependencies, and debugging issues that arise at the JavaScript-native boundary.

Finally, the total memory footprint of Path B must be considered. While the MobileBERT model itself is relatively small at approximately 25MB, this is only one component of the solution [[1](https://arxiv.org/html/2601.03290v1)]. The inference engine, such as ONNX Runtime, also carries a memory overhead. The exact size of the runtime binaries and their peak RAM usage during inference are not specified in the provided sources and would need to be determined through profiling on target devices. The combined memory impact of the model file plus the runtime must be kept under the project's strict <50MB (ideally <25MB) limit. This means that simply adding a ~25MB model file might not be sufficient; the entire on-device inference package must be optimized and measured. Quantization techniques, such as converting model weights from 32-bit floating-point (FP32) to 8-bit integer (INT8), can reduce the model size by up to 4x and improve latency, but this may come at the cost of a slight accuracy degradation (typically 0.5-1.2%) [[1](https://arxiv.org/html/2601.03290v1)]. Careful experimentation with quantization levels will be necessary to find the right balance between size, speed, and accuracy for the deployed model. In conclusion, while technically feasible, the integration of Path B is a non-trivial engineering undertaking. It requires expertise in both the React Native ecosystem and native mobile development, careful planning for dependency management and builds, and rigorous performance profiling to ensure all constraints are met. The complexity of this path stands in stark contrast to the immediate feasibility of Path A, representing a major strategic decision about the project's resource allocation.

## Fine-Tuning and Data Requirements for Custom Intent Classification

The promise of high accuracy (>95%) in Path B hinges entirely on the ability to fine-tune a lightweight model on a relevant dataset of the 20+ custom intents. Unlike Path A, which relies on the latent capabilities of a pre-trained model, Path B requires a proactive, data-centric approach to model development. This process involves collecting and labeling a substantial corpus of user utterances for each intent class, training a model on this data, and then exporting the resulting "fine-tuned" model for on-device deployment. This is a significant undertaking that requires both data science expertise and a well-defined data collection strategy. The feasibility of this path is therefore directly tied to the availability of quality labeled data and the team's capacity to execute the fine-tuning process effectively.

The first and most critical prerequisite for fine-tuning is a labeled dataset. The current regex-based system, while inaccurate, likely contains a wealth of user interactions that can be repurposed. The first step would be to collect a representative sample of user inputs that correspond to each of the 20+ intents. For each input, a human annotator would need to assign the correct intent label. The quantity of data required is a key consideration. While large language models can sometimes generalize from few-shot examples, fine-tuning smaller models like MobileBERT or DistilBERT typically benefits from a more substantial dataset. A rule of thumb is to have hundreds of examples per class to ensure the model learns robust representations and avoids overfitting. In scenarios where labeled data is scarce, which is often the case when starting a new classification task, data augmentation becomes an essential technique [[18](https://www.researchgate.net/publication/361055801_Data_Augmentation_for_Intent_Classification_with_Off-the-shelf_Large_Language_Models)]. Augmentation methods can synthetically expand the dataset by applying transformations to existing examples, such as synonym replacement, back-translation, or using techniques described in frameworks like IntentGPT, which uses LLMs to generate new examples with minimal labeled data [[25](https://arxiv.org/html/2411.10670v1)]. This data preparation phase is a non-negotiable step and represents a significant upfront effort before any model training can begin.

Once a labeled dataset is prepared, the fine-tuning process can commence. This is typically done using established deep learning frameworks. The Hugging Face `transformers` library is a popular choice, as it provides pre-trained models like MobileBERT, DistilBERT, and TinyBERT, along with convenient APIs for tokenization, training, and evaluation [[31](https://www.kaggle.com/code/goutham794/distill-bert-fine-tuning-huggingface-and-pytorch)]. Fine-tuning can be performed using either PyTorch or TensorFlow. For instance, one source details a workflow for fine-tuning a `TFDistilBertForSequenceClassification` model using TensorFlow, involving the creation of a `tf.data.Dataset` from tokenized encodings and labels, followed by compilation and fitting of the model using an Adam optimizer [[32](https://stackoverflow.com/questions/64500833/fine-tuning-huggingface-distilbert-for-multi-class-classification-on-custom-data)]. A typical training regimen might involve a learning rate of 5e-5 and training for a few epochs (e.g., 3) to adapt the model to the new task without overwriting the general knowledge it has already learned [[32](https://stackoverflow.com/questions/64500833/fine-tuning-huggingface-distilbert-for-multi-class-classification-on-custom-data)]. A common pitfall in this process is incorrect data handling, such as misconfiguring the dataset batching or tokenization, which can lead to unexpected output shapes and poor performance [[32](https://stackoverflow.com/questions/64500833/fine-tuning-huggingface-distilbert-for-multi-class-classification-on-custom-data)]. Careful attention to the training loop and evaluation metrics is crucial.

To make the fine-tuning process more computationally efficient, especially when working with limited resources, Parameter-Efficient Fine-Tuning (PEFT) techniques can be employed. Low-Rank Adaptation (LoRA) is a prominent PEFT method that freezes the vast majority of the model's original weights and injects small, trainable "rank decomposition" adapters into the layers of the Transformer [[30](https://www.academia.edu/125096234/INTENT_RECOGNITION_USING_DISTILBERT_AND_LANGUAGE_MODELS)]. During training, only these small adapter matrices are updated, drastically reducing the number of trainable parameters and the memory required for the backward pass. This allows for faster training and lower memory consumption, making it easier to experiment with different model architectures and hyperparameters. Using LoRA or similar techniques can significantly lower the barrier to entry for fine-tuning larger lightweight models. The research highlighted that DistilBERT, when fine-tuned, showed faster training and inference times compared to larger models, partly due to its smaller parameter count, but PEFT methods can further enhance this efficiency [[30](https://www.academia.edu/125096234/INTENT_RECOGNITION_USING_DISTILBERT_AND_LANGUAGE_MODELS)].

In summary, the path to a high-performing classifier via Path B is paved with data. The feasibility of this entire approach is contingent on the project's willingness to invest in the data collection and labeling process. Without a solid, well-curated dataset, the promise of >95% accuracy cannot be realized. Once the data is ready, the fine-tuning process itself is well-understood and supported by mature open-source tools. By leveraging PEFT methods like LoRA, the team can mitigate some of the computational costs associated with this process. The successful execution of these steps will yield a model that is highly specialized for the Lyla application's intent set, offering a level of accuracy and reliability that is difficult to achieve with a general-purpose model via prompting alone. This data-driven approach transforms intent classification from a black-box guessing game into a predictable, engineering-driven outcome.

## Performance Benchmarking and Hardware Considerations

A thorough evaluation of the two proposed paths requires a detailed look at their expected performance characteristics, particularly concerning latency and memory footprint, and how these metrics are influenced by the underlying hardware and software stack. The goal is to determine if either Path A or Path B can consistently meet the project's demanding targets of <50ms latency and a <50MB (ideally <25MB) model size. The provided research offers valuable benchmark data and insights into hardware utilization patterns on edge devices, which are crucial for making an informed decision. The interplay between model architecture, optimization techniques like quantization, and the efficiency of the inference engine running on a device's CPU or GPU ultimately determines real-world performance.

For Path B, the MobileBERT model serves as a primary reference point. According to one study, MobileBERT, with 25.3 million parameters, achieves a GLUE score of 77.7 and an SQuAD v1.1 F1 score of 90.3 [[1](https://arxiv.org/html/2601.03290v1)]. Its benchmarked inference latency on a Pixel 4 phone was 62ms. It is important to note that this latency figure was obtained on a full-precision FP32 model, which is less efficient than the 8-bit integer (INT8) quantized versions commonly used for on-device deployment. INT8 quantization can reduce model size by up to 4x and significantly improve inference speed, though it may introduce a minor accuracy loss of 0.5-1.2% [[1](https://arxiv.org/html/2601.03290v1)]. When considering the Hermes engine, which is Apple's JavaScript VM for React Native, performance can vary between different inference engines or between different versions and installation dates [[5](https://arxiv.org/pdf/2508.18255)]. This suggests that the 62ms latency figure is a baseline that can likely be improved upon with modern hardware and optimizations. Another study on a phishing detection system using MobileBERT reported a median prediction time of 0.39 seconds (390ms) per prediction, but this appears contradictory and may refer to a different implementation or a worst-case scenario, as it is significantly higher than the 62ms figure [[10](https://arxiv.org/html/2509.10858v1)]. A more optimistic projection from the same research indicates that for interactive UI applications, models like MobileBERT and TinyBERT-6 can achieve latencies low enough to provide a 100ms response time, which is well within the project's <50ms target [[1](https://arxiv.org/html/2601.03290v1)].

The hardware utilization of these lightweight models is a key factor in their on-device performance. Research reveals that for interactive applications, models with 15-40 million parameters achieve optimal hardware utilization (60-75%) on typical edge devices [[1](https://arxiv.org/html/2601.03290v1)]. Models larger than this threshold often become memory-bandwidth limited, meaning that the speed at which data can be fetched from memory becomes the bottleneck, rather than the raw computational power of the processor. This explains why MobileBERT (25.3M parameters) and the larger TinyBERT-6 (67M parameters) can exhibit similar latencies despite their significant difference in parameter count; both are operating in a regime where memory access is a dominant factor [[1](https://arxiv.org/html/2601.03290v1)]. This insight is critical: it suggests that simply choosing the smallest model is not always the best strategy. There is an optimal size range for maximizing computational throughput on mobile hardware. Both MobileBERT and TinyBERT-4 fall squarely within this optimal range, making them excellent candidates for achieving high performance on modern smartphones.

In stark contrast, FastText offers a different paradigm of extreme efficiency. One study reported an inference latency of just 0.31 milliseconds per document for a FastText-based model, which is orders of magnitude faster than any of the transformer-based models discussed [[2](https://arxiv.org/html/2604.17674v1)]. This incredible speed stems from its linear classification algorithm, which is far less computationally intensive than the self-attention mechanisms of a Transformer. This raw speed advantage means that a FastText model could easily meet and likely exceed the <50ms latency target, even on lower-end devices. The memory footprint is equally impressive, with the FastText model requiring only 5.1 million parameters and 1.3 GB of GPU memory in one experiment, dwarfed by the 110 million parameters and 5.8 GB of GPU memory required by a BERT model for a comparable task [[2](https://arxiv.org/html/2604.17674v1)]. This combination of sub-millisecond latency and a minuscule memory footprint makes FastText an exceptionally attractive option, challenging the dominance of Transformer models for certain classification tasks.

From a hardware perspective, the minimum viable hardware for these models includes a modern mobile GPU or a CPU with a capable AI accelerator. The research suggests that for interactive UI applications, hardware with at least 10 TOPS (Trillion Operations Per Second) of AI acceleration or a modern mobile GPU is required to comfortably run models like MobileBERT and TinyBERT-6 [[1](https://arxiv.org/html/2601.03290v1)]. Most flagship and many mid-range smartphones released in recent years meet or exceed this specification. The Hermes engine, being a Just-In-Time (JIT) compiler, translates JavaScript/TypeScript into more efficient machine code, which can help accelerate the execution of the inference logic and data handling in the React Native layer, complementing the work of the native inference engine [[5](https://arxiv.org/pdf/2508.18255)]. However, the bulk of the computational work will be handled by the native module of the chosen inference engine (e.g., ONNX Runtime).

In conclusion, the performance analysis strongly favors Path B over Path A. While Path A's latency is estimated to be in the 50-100ms range, which is borderline for a smooth user experience, Path B offers a clear path to achieving latencies well under the 50ms target. MobileBERT, when properly quantized and run on modern hardware, is projected to be very fast. However, FastText emerges as a standout performer, offering not just fast but exceptionally fast inference times that are unlikely to be a bottleneck on any target device. In terms of memory, both MobileBERT (~25MB) and FastText (~5MB) are well within the project's constraints, although the total memory impact of Path B must account for the inference engine itself. The hardware considerations indicate that the target devices are capable of running these models effectively, provided they are properly optimized. The key takeaway is that Path B provides a technologically sound path to meeting all performance targets, whereas Path A's performance is uncertain and potentially suboptimal.

## Strategic Recommendation and Implementation Roadmap

After a comprehensive analysis of the two proposed paths, a clear strategic recommendation emerges. The optimal course of action is not a simple choice between MobileBERT and the existing 350M model, but rather a phased, risk-managed approach that prioritizes validating the lowest-effort option first before committing to a more complex solution. This strategy balances the competing demands of accuracy, latency, memory, and implementation complexity. The evidence strongly suggests that Path A, leveraging the existing LFM2.5-350M model via prompting, should be treated as the primary, immediate solution to be validated. Only if it demonstrably fails to meet the critical performance thresholds should the team proceed to the higher-complexity but potentially higher-performing Path B. This roadmap minimizes initial engineering overhead and allows for an evidence-based decision, de-risking the project's timeline and resource allocation.

The first and most critical action is to immediately implement a proof-of-concept for Path A. This is a low-cost, high-value activity that will provide definitive data on the viability of the existing model for the intent classification task. The engineering effort should focus on creating a robust and optimized prompt template. This involves carefully crafting the instruction, listing the 20+ intent names, and defining the input format. It is crucial to test this prototype across a wide and representative sample of user utterances for each intent, including clear cases, ambiguous cases, and out-of-scope queries. The primary metrics to measure are accuracy and latency. Accuracy should be evaluated against a manually labeled gold-standard test set to get a true sense of performance. Latency should be measured on a variety of target iOS and Android devices to understand the real-world user experience. If Path A successfully meets or exceeds the >95% accuracy and <50ms latency targets, it should be adopted for production immediately. Its advantages—zero memory overhead, no additional native dependencies, and rapid implementation—are so significant that it represents the ideal solution if it performs adequately. In this scenario, the research phase is complete, and development resources can be focused on refining the prompt, expanding the intent set, and improving the overall user experience.

However, given the inherent unpredictability of zero-shot prompting, it is more likely that Path A will fall short on either accuracy or latency. If the validation of Path A fails to meet the project's stringent requirements, the team should trigger the transition to Phase 2: pursuing Path B. This phase involves the full integration of a dedicated lightweight classifier. The very first step in this phase must be a focused effort on data collection and preparation. Since Path B's success is contingent on fine-tuning, the team must prioritize creating a high-quality, labeled dataset for the 20+ intents. This is a non-trivial data science task that should be undertaken with care. Once the dataset is ready, the next step is to select a model candidate. Based on the analysis, it is recommended to start with FastText. Its combination of high accuracy potential, extreme speed, and low resource consumption makes it an ideal candidate to serve as a powerful baseline. The team should fine-tune a FastText model on the collected dataset and evaluate its performance. If the FastText model meets the >95% accuracy target, it should be integrated into the app. Its simplicity and efficiency would make it the preferred choice over more complex Transformer models.

If the FastText model fails to reach the required accuracy threshold, the team should then proceed to integrate a Transformer-based model, with MobileBERT being the prime candidate. This involves the more complex steps of: 1) fine-tuning MobileBERT on the labeled dataset using a framework like Hugging Face and PyTorch; 2) exporting the fine-tuned model to the ONNX format; and 3) integrating the `onnxruntime-react-native` library into the Expo project, likely using EAS Build to manage the native dependencies. Throughout this process, continuous profiling of both latency and memory footprint is essential to ensure that the final solution adheres to the project's constraints. This tiered approach ensures that the team does not prematurely commit to the highest level of complexity. They first validate a simple, elegant solution (Path A), then attempt a highly efficient alternative (FastText), and finally resort to a state-of-the-art but complex solution (MobileBERT) only when all else has failed.

In conclusion, the decision between Path A and Path B is a strategic one about risk and resource allocation. Path A is the safe, low-risk bet that offers immediate value with known limitations. Path B is the high-risk, high-reward endeavor that promises superior performance but at the cost of significant engineering effort. The recommended phased roadmap mitigates this risk by front-loading the validation of the simplest possible solution. This approach ensures that the team invests in the complexity of Path B only when it is truly necessary to meet the project's goals. By starting with Path A, experimenting with FastText, and using MobileBERT as a final fallback, the team can navigate the trade-offs intelligently and arrive at a solution that is both technically excellent and pragmatically achievable.
